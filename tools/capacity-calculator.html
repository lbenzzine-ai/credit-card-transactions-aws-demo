Before you present this savings number

  • 1
    Only I/O-wait time benefits — not CPU-bound work. Virtual threads are cheap because they park during I/O waits. If part of your request is real computation (serialization, crypto, validation), that portion is still bound by CPU cores per instance, not thread count.
  • 2
    You've concentrated load, not eliminated it. Whether spread across 25 instances or 3, your downstream dependency (fraud-check, DB, etc.) still receives the same total concurrent calls — now from fewer source hosts, which can trip its own connection limits or rate limits.
  • 3
    Blast radius gets worse with fewer instances. Losing 1 of 25 costs ~4% of capacity. Losing 1 of 3 costs ~33%, instantly. Provision above the bare mathematical minimum specifically to keep this sane.
  • 4
    Per-instance cost goes up. Holding hundreds of concurrent requests' worth of heap needs meaningfully more memory per box. This is "fewer, bigger, somewhat-more-expensive-each" instances, not the same box with fewer copies.
  • 5
    Watch for virtual thread pinning. synchronized blocks or certain legacy JDBC drivers can pin a virtual thread to its carrier thread during a blocking call, silently defeating the benefit. Validate against your actual dependency stack.
  • 6
    This is a planning estimate, not a commitment. The real number comes from load testing under realistic conditions — especially GC behavior at much higher heap utilization per instance.

Please note

Moving to virtual threads can meaningfully reduce instance count — potentially 8-10x fewer for the same throughput and latency target, since capacity is no longer bounded by thread pool size. Before treating that as a final number, validate three things: whether downstream dependencies can absorb the same concurrent load now concentrated on fewer hosts, whether any part of the request path is CPU-bound rather than I/O-bound, and provision above the bare-minimum instance count for redundancy and blast-radius reasons. The cost savings are real, but the final figure should come from load testing, not from this calculator alone.
Little's Law: concurrency = throughput × latency. Capacity per instance = thread pool size (platform threads) or heap ÷ memory-per-request (virtual threads / async).
